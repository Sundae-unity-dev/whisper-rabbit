# 음성 회의 녹음 정리

음성/영상 녹음 파일(`.mp3` / `.m4a` / `.wav` / `.mp4` / `.mov` / `.opus` / `.flac` 등 ffmpeg가 디코드 가능한 포맷 전부)을 받아, faster-whisper로 받아쓰기한 뒤 회의 내용을 **요약 + 논의 내용 + 문제점 + 해결방안** 4섹션으로 정리한 단일 회의록 docx 를 만든다 (맨 앞 페이지가 한 눈에 읽히는 요약, 그 뒤로 본문 3섹션).

`.mp4` / `.mov` 같은 영상 컨테이너도 ffmpeg가 오디오 트랙만 자동 추출하므로 별도 변환 없이 그대로 넘기면 된다.

기본 저장 위치는 Desktop.

## 사용법

```
/회의녹음정리 <오디오파일경로>
              [--model small|medium|large-v3]
              [--team 팀이름]
              [--date YYYY-MM-DD]
              [--meeting-type general|scrum|review|retro]
              [--meta minimal|full]
              [--no-toc] [--no-context] [--no-cover]
```

기본값: `--model small --meeting-type general --meta minimal` (목차·표지·회의 맥락 모두 포함).

예시:
- `/회의녹음정리 "C:\Users\21-01-00131\Downloads\팀2_0513_스크럼.m4a"` — 기본 옵션
- `/회의녹음정리 "C:\...\meeting.mp3" --model medium --meeting-type scrum` — 스크럼 회의 + medium 모델
- `/회의녹음정리 "C:\...\회고.mp4" --meeting-type retro --meta full` — 회고 + 메타 표 6행
- `/회의녹음정리 "C:\...\리뷰.m4a" --meeting-type review --no-context` — 리뷰 + 회의 맥락 섹션 제외
- 경로에 공백이 있으면 반드시 큰따옴표.

### 커스텀 옵션 의미

| 옵션 | 효과 |
|---|---|
| `--meeting-type general` (기본) | 본문 3섹션: **1. 논의 내용 / 2. 문제점 / 3. 해결방안** |
| `--meeting-type scrum` | **1. 어제 한 일 / 2. 오늘 할 일 / 3. 블로커·이슈** |
| `--meeting-type review` | **1. 리뷰 의견 / 2. 개선 포인트 / 3. 후속 액션** |
| `--meeting-type retro` | **1. Keep (잘된 것) / 2. Problem (개선 필요) / 3. Try (시도)** |
| `--meta minimal` (기본) | 요약 메타 표 3행: 녹음 길이 / 분류 항목 / 액션 아이템 |
| `--meta full` | 위 3행 + 받아쓰기 / 모델·장치 / audio_sha1 (총 6행) |
| `--no-toc` | 목차 페이지 생략 |
| `--no-cover` | 표지 페이지 생략 |
| `--no-context` | 요약 페이지의 "회의 맥락" 서브섹션 생략 |

인자: $ARGUMENTS

## 🚨 배치 격리 원칙

이번 호출에서 주어진 오디오만 처리한다. 이전 대화의 다른 회의·문서 내용이 결과에 섞이지 않게, 작업용 임시 파일은 매 호출마다 새로 분리한다.

## 작업 순서

### 1단계: 입력 파싱

1. $ARGUMENTS에서 오디오 파일 경로를 추출. 옵션(`--model`, `--team`, `--date`)도 파싱.
2. 파일 존재 확인. 없으면 사용자에게 경로 재확인 요청.
3. 파일명에서 **팀명·날짜**를 추정 가능하면 추정 (예: `팀2_0513_스크럼.m4a` → 팀=팀2, 날짜=2026-05-13). 추정 실패 시 `--team`/`--date` 옵션이나 사용자 확인.
4. 사용자에게 "이번 배치: <파일명>, 팀=<팀>, 날짜=<날짜>, 모델=<small|medium|...>로 진행합니다" 한 줄 안내.

### 2단계: 모델 추천 → whisper-rabbit 실행

#### 2-1. 사전 점검 (필수)

받아쓰기 시작 전 한 번 `whisper_rabbit.recommend` 를 호출해 오디오 길이와 추천 모델·예상 시간을 사용자에게 안내한다. 이 한 단계로 "왜 안 끝나지" 체감을 차단한다.

```powershell
python -m whisper_rabbit.recommend "<audio>"
```

출력 JSON 의 `duration_minutes` / `recommended_model` / `estimated_minutes` / `reason` / `notes` 를 사용자에게 한 줄로 보고하고 진행 의사를 확인. 사용자가 별도 `--model X` 를 주지 않았다면 `recommended_model` 을 그대로 사용.

#### 2-2. 받아쓰기 실행

```powershell
python -m whisper_rabbit "<audio>" --model <model> --lang ko --formats txt,json
```

- 첫 실행 시 모델 다운로드 시간이 걸리므로 Bash `run_in_background`로 띄우고 Monitor로 진행 상황을 추적할 것.
- 출력 파일:
  - `<audio>.txt` — `[HH:MM:SS] 문장` 줄 단위 transcript
  - `<audio>.json` — segments + 메타데이터 (audio_sha1, duration, model, device, RTF 등)
- 표준출력의 결과 JSON 에서 transcript 경로를 읽어 다음 단계로.

### 3단계: transcript 읽기 및 정리 — 세 가지 핵심 구조

`<audio>.txt` 파일을 Read 도구로 읽고, 아래 3개 섹션으로 재구성한다.

#### 1. 논의 내용 (이번 회의에서 오간 이야기)
- 회의에서 다뤄진 주제·발언·결정·공유 내용.
- 시간 순서가 중요한 경우 자연스러운 흐름을 유지하고, 같은 주제는 묶는다.
- "누가 말했는지"는 transcript에서 확정 불가하므로 **발언자명은 적지 않는다**. 의제·내용만 남긴다.

#### 2. 문제점 (회의에서 제기된 문제·이슈·리스크)
- 명시적으로 문제로 거론된 항목. "잘 안 된다", "막혔다", "오류", "리스크", "걱정", "이해 안 됨" 등.
- 각 항목은 한두 줄로 압축. 가능하면 **주제 키워드**를 앞에 단다 (예: `[빌드 환경 차이]`).

#### 3. 해결방안 (스크럼/논의에서 나온 대응책·액션·합의)
- 문제점에 대응되는 해결책·후속 액션·결정 사항.
- 가능하면 2번 항목과 1:1 대응되도록 같은 주제 키워드로 묶는다.
- 아직 해결책이 나오지 않은 문제는 "**미해결 / 후속 논의 필요**" 표시.

### 정리 시 규칙 (엄수)

- **하이픈 `-` 불릿** 고정. 이모지·`◆` 소제목 금지.
- **체언형 종결** (`~완료`, `~수정`, `~정리`, `~필요`, `~예정` 등). 한 불릿 = 한 문장.
- **섹션당 5~10개 불릿 목표**. 비슷한 항목은 병합.
- **발언자명·`(이름)` 표기 금지** (사용자 요청: 화자 구분은 나중에 수동 정리).
- transcript의 오타·받아쓰기 오류는 문맥에 맞게 교정하되 의미를 바꾸지 않는다.
- 잡담·인사·의미 없는 추임새는 제외.
- 영문 기술 용어는 그대로 노출 (JSON, API, Dictionary 등).

### 4단계: 회의록 docx 생성 (whisper_rabbit.report 모듈 호출)

docx 구조·python-docx 구현·TOC·페이지 번호·익명화 검증은 **모두 `whisper_rabbit.report` 패키지가 처리**한다. 슬래시 커맨드는 분류 결과를 JSON 으로 만들어 모듈에 넘기는 일까지만 책임진다.

#### 4-1. 데이터 JSON 작성

배치 임시 폴더(`$env:TEMP\meeting_batch_<ts>\data.json`)에 다음 형식의 JSON 을 쓴다. 전체 스키마는 `python -m whisper_rabbit.report --print-schema` 로 확인 가능.

```json
{
  "team": "<팀명>",
  "date": "<YYYY-MM-DD>",
  "audio_file": "meeting.mp4",
  "audio_duration": "13:23",
  "audio_sha1": "abc...",
  "model_label": "small / cpu / int8",
  "transcribe_time": "3:44 (RTF 0.28x)",

  "tldr": "회의 결론을 1~3 문장(100~250자) 으로 압축한 단락.",
  "keywords": ["키워드1", "키워드2", "키워드3"],

  "key_points": [
    {"title": "포인트 제목", "body": "한 줄 본문", "sub": "선택 부연(없으면 생략)"}
  ],
  "decisions": [
    {"title": "결정 제목", "body": "내용 + 이유/배경"}
  ],
  "actions": [
    {"number": "1", "action": "할 일", "owner": "?", "due": "?", "status": "신규"}
  ],
  "unresolved": [
    {"title": "이슈 제목", "body": "사유 + 다음 결정 시점/방법"}
  ],
  "context": ["출결·휴가·일정 등 의제 외 보조 정보"],

  "body": [
    [{"name": null, "items": ["섹션 1 불릿"]}],
    [{"name": "주제 키워드", "items": ["섹션 2 불릿"]}],
    [{"name": "주제 키워드", "items": ["섹션 3 불릿"]}]
  ]
}
```

`body` 의 각 섹션은 `Topic[]` 배열이다. `name=null` 이면 헤더 없이 평이 불릿, 문자열이면 `[주제]` 블록 헤더로 묶인다.

#### 4-2. 모듈 호출

```powershell
python -m whisper_rabbit.report `
  --input "$env:TEMP\meeting_batch_<ts>\data.json" `
  --meeting-type <type> --meta <mode> [토글들]
```

옵션 매핑:

| 슬래시 인자 | 모듈 옵션 |
|---|---|
| `--meeting-type general\|scrum\|review\|retro` | `--meeting-type ...` |
| `--meta minimal\|full` | `--meta ...` |
| `--no-toc` / `--no-cover` / `--no-context` | 동일 |
| `--output <path>` (선택) | `--output <path>` — 미지정 시 `~\Desktop\<팀>_회의녹음정리_<날짜>.docx` |

모듈은 stdout 에 다음 형식의 결과 JSON 을 출력한다. 슬래시 커맨드는 이 결과를 사용자에게 그대로 보고하면 된다.

```json
{
  "docx": "...절대경로...",
  "docx_size_kb": 40.8,
  "section_counts": { ... },
  "verification": "passed"
}
```

#### 4-3. 회의 유형별 본문 매트릭스 (`--meeting-type`)

| 유형 | ■ 1 | ■ 2 | ■ 3 |
|---|---|---|---|
| `general` (기본) | 논의 내용 | 문제점 | 해결방안 |
| `scrum` | 어제 한 일 | 오늘 할 일 | 블로커·이슈 |
| `review` | 리뷰 의견 | 개선 포인트 | 후속 액션 |
| `retro` | Keep (잘된 것) | Problem (개선 필요) | Try (시도) |

요약 페이지 메타 표의 "분류 항목" 도 위 섹션 이름을 그대로 사용한다.

#### 4-4. 모듈이 보장하는 것 (별도로 신경 쓸 필요 없음)

- 표지 → 목차 → ■0 회의 요약 → ■1 / ■2 / ■3 본문 순으로 **섹션마다 새 페이지에서 시작** (`w:pageBreakBefore`)
- 모든 페이지 가운데 정렬 `PAGE` 필드 푸터
- Word 가 처음 열 때 TOC·PAGE 자동 채움 (`w:updateFields=true`)
- 모든 텍스트에 맑은 고딕 폰트 (run 단위로 `w:eastAsia/w:ascii/w:hAnsi` 적용)
- 익명화 검증 — `슈가/다솔림/유지님/지혜님` 등 잔존 시 저장 거부
- 출력 파일이 Word 로 열려 있을 때 `_v2`, `_v3` 증분 저장

#### 요약 페이지 작성 규칙 (엄수)

- **TL;DR**: 1~3 문장, 100~250자. 결론·핵심 화두를 압축. 단순 의제 나열 금지. `Intense Quote` 스타일 또는 회색 배경/좌측 컬러 바로 강조.
- **메타 표**: `<audio>.json` 메타에서 그대로 가져온다. 추정 금지. 표 스타일은 `Light Grid Accent 1` 또는 동등한 격자.
- **핵심 키워드**: 5~7개. 한국어/영문 가리지 않음, 사람 이름 금지. 가운뎃점(`·`)으로 구분한 한 줄.
- **핵심 포인트**: 6~10개. 한 줄 본문 + 필요 시 1줄 부연(들여쓰기 단락). 부연은 *왜 중요한가* / *후속 영향*에 집중.
- **결정 사항**: 번호 매김 단락. `**제목** — 내용 + 이유` 형식. 확정 안 된 항목은 미해결 이슈로.
- **액션 아이템**: docx 표 5컬럼(`# / 액션 / 담당 / 기한 / 상태`). 담당·기한 미확정은 `?`. 상태는 한 단어(`신규` / `후속 논의 필요` / `진행 중` 등).
- **미해결 이슈**: 사유 + 다음 결정 시점/방법 함께. "후속 논의 필요" 만 적지 않는다.
- **회의 맥락**: 출결·휴가·행정 공지 등 본 의제와 별개의 보조 정보. 핵심 포인트에 섞이지 않게 분리.
- **발언자명 금지**: 사람 이름은 어디에도 적지 않는다. 액션 담당란도 추정으로 채우지 않는다.

#### STT 오류 / 익명화 / 사실 검증 (엄수 — 과거 실수 재발 방지)

**과거 실수 사례와 회피 규칙**:

1. **한국어 STT 오인식은 자연스럽게 교정** — receive 가 transcript 그대로 옮기지 않는다. 받아쓰기는 동음이의어를 자주 잘못 표기한다.
   - 사례: 음성의 "휴가" 가 transcript 에 `슈가` 로 표기. 회의록에 `슈가(휴가)` 형태로 (원어)병기한 게 어색함.
   - 규칙: **의미를 바꾸지 않는 선에서 문맥에 맞는 정확한 한국어로 풀어쓴다.** `슈가(휴가)`, `슈가 / 휴가`, `(원음: 슈가)` 같은 병기 모두 금지. 회의록은 의미 정리물이지 받아쓰기 재현물이 아니다.

2. **역할·관계 단정 금지** — transcript 에 음성으로 명시되지 않은 역할명을 추측으로 붙이지 않는다.
   - 사례: 화자가 "제가 어제 자리를 비웠습니다" 라고 말했는데 회의록에 "어제 **코치** 부재" 로 단정. 실제로는 코치와 진행자가 별개 인물.
   - 규칙: 부재자·책임자가 누구인지 음성에서 확정 불가하면 `코치` / `선생님` / `멘토` / `매니저` 같은 직책명을 적지 말고 **`회의 진행자` / `운영자` / `참가자` 같은 익명 호칭**으로 적는다. 인물 이름이 transcript 에 *호명*되어 있더라도, 그 인물이 어떤 행위를 했는지가 분명하지 않으면 행위 주체로 등장시키지 않는다.

3. **사람 이름은 어떤 형태든 일괄 제거** — 본명·닉네임·`~님` 호칭 모두 익명화.
   - 사례: `다솔림 병가, 유지님 휴가` 그대로 옮긴 것.
   - 규칙: `동료 1명 병가, 1명 휴가` 같이 익명화. **docx 모든 섹션(요약·논의·문제점·해결방안·회의 맥락·액션 표 담당란) 에 동일하게 적용**한다. 액션 담당란은 항상 `?`.

- [ ] 역할 단정 잔존 — `코치`, `선생님`, `멘토`, `매니저` 등 직책명을 transcript 에 호명되지 않은 인물에 붙이지 않았는가?

- [ ] 사실관계 잔존 — transcript 에 *명시적으로 등장하지 않는* 사실을 추정으로 보태지 않았는가? 의심 항목은 `[분류 확인 필요]` 표기.

#### 스타일 규칙 (엄수)

- **한글 폰트**: 맑은 고딕. 모든 run에 `w:eastAsia`, `w:ascii`, `w:hAnsi` 3가지로 적용. Normal 스타일도 동일.
- **페이지 브레이크**: `WD_BREAK.PAGE` 사용. 표지/요약/3섹션 사이에 강제 삽입.
- **색상 계층**:
  - 표지 제목 18pt, 굵게, 가운데 정렬
  - 섹션 제목(`■ N.`) 15pt, 진한 남색 `RGBColor(0x1F, 0x4E, 0x79)`, 굵게
  - 요약 페이지 내 하위 헤더(`[핵심 포인트]` 등) 12pt, 밝은 파랑 `RGBColor(0x2E, 0x75, 0xB6)`, 굵게
  - `[주제]` 블록 헤더 (2·3번 섹션) 12pt, 밝은 파랑 동일
  - 본문 불릿 `List Bullet`, 번호 `List Number`
- **표**: `add_table()` 사용. 헤더 행은 굵게 + 밝은 회색 배경(`D9D9D9`) 셀 음영. 모든 셀 폰트도 맑은 고딕으로 일관.

### 5단계: 저장 및 결과 보고

이번 호출의 산출물은 **단일 회의록 docx 한 개**다 (`.summary.md` 는 더 이상 만들지 않는다 — 요약은 docx 의 0번 섹션에 통합).

- **파일명**: `<팀명>_회의녹음정리_<날짜>.docx` (예: `팀2_회의녹음정리_2026-05-13.docx`)
- **저장 위치**: `C:\Users\21-01-00131\Desktop\`
- 같은 이름이 Word 로 열려 있어 `PermissionError` 발생 시 `_v2`, `_v3` 로 증분 저장.
- **transcript txt/json 은 받아쓰기 출력 위치(`<audio>` 같은 폴더)에 그대로 둔다.** 사용자가 나중에 원문 확인·화자 보정에 사용.
- **PDF가 필요할 때**: Word 로 열어 `파일 → 내보내기 → PDF/XPS 만들기`. 별도 의존성 없이 가능.

#### 결과 보고 (간결히)

- 생성된 docx 절대경로 + 크기
- transcript txt/json 절대경로
- 녹음 길이, 처리 소요 시간, RTF, 사용한 모델 / device / compute_type (json 메타에서 읽어 표시)
- 회의록 섹션별 항목 개수 (요약·핵심 포인트 N / 결정 M / 액션 K / 미해결 L / 논의 P / 문제점 Q / 해결방안 R)
- 분류가 모호해 `[분류 확인 필요]`로 표기한 항목이 있다면 목록 안내

## 주의사항

- transcript는 받아쓰기 결과라 **오타·문장 경계 오류**가 있을 수 있다. 정리할 때는 의미 단위로 묶어서 의도가 명확해지도록 다듬되, **사실관계를 추측으로 보태지 않는다**. 들리지 않는 부분이 분류에 핵심이면 `[분류 확인 필요]`로 남긴다.
- transcript에서 발언자를 추정하지 않는다. (사용자 요청: 누가 말했는지는 나중에 사람이 직접 정리)
- 이전 회의·다른 팀 내용은 어떤 경로로도 참조하지 않는다.
- 첫 실행 시 Whisper 모델 다운로드로 몇 분~10분 이상 걸릴 수 있음을 사용자에게 미리 안내.
- 매우 긴 회의(2시간+)는 받아쓰기만 30분+ 걸릴 수 있다. `run_in_background`로 띄우고 Monitor로 진행 상황을 사용자에게 보여준다.
