# 음성 회의 녹음 정리

음성/영상 녹음 파일(`.mp3` / `.m4a` / `.wav` / `.mp4` / `.mov` / `.opus` / `.flac` 등 ffmpeg가 디코드 가능한 포맷 전부)을 받아, faster-whisper로 받아쓰기한 뒤 회의 내용을 **논의 내용 / 문제점 / 해결방안** 중심으로 정리한 통합 회의록 docx + 한 화면 요약본(.summary.md)을 만든다.

`.mp4` / `.mov` 같은 영상 컨테이너도 ffmpeg가 오디오 트랙만 자동 추출하므로 별도 변환 없이 그대로 넘기면 된다.

기본 저장 위치는 Desktop.

## 사용법

```
/회의녹음정리 <오디오파일경로> [--model small|medium|large-v3] [--team 팀이름] [--date YYYY-MM-DD]
```

예시:
- `/회의녹음정리 "C:\Users\21-01-00131\Downloads\팀2_0513_스크럼.m4a"`
- `/회의녹음정리 "C:\Users\21-01-00131\Downloads\meeting.mp3" --model medium`
- 경로에 공백이 있으면 반드시 큰따옴표.

인자: $ARGUMENTS

## 🚨 배치 격리 원칙

이번 호출에서 주어진 오디오만 처리한다. 이전 대화의 다른 회의·문서 내용이 결과에 섞이지 않게, 작업용 임시 파일은 매 호출마다 새로 분리한다.

## 작업 순서

### 1단계: 입력 파싱

1. $ARGUMENTS에서 오디오 파일 경로를 추출. 옵션(`--model`, `--team`, `--date`)도 파싱.
2. 파일 존재 확인. 없으면 사용자에게 경로 재확인 요청.
3. 파일명에서 **팀명·날짜**를 추정 가능하면 추정 (예: `팀2_0513_스크럼.m4a` → 팀=팀2, 날짜=2026-05-13). 추정 실패 시 `--team`/`--date` 옵션이나 사용자 확인.
4. 사용자에게 "이번 배치: <파일명>, 팀=<팀>, 날짜=<날짜>, 모델=<small|medium|...>로 진행합니다" 한 줄 안내.

### 2단계: whisper-rabbit 실행

설치된 패키지를 모듈로 호출한다 (저장소 루트가 PYTHONPATH 에 있거나 `pip install -e .` 된 상태 기준).

```powershell
python -m whisper_rabbit "<audio>" --model <model> --lang ko --formats txt,json
```

- 기본 모델은 `small` (한국어 적당·빠름). 음질이 나쁘거나 정확도 우선이면 `medium` 권장.
- 첫 실행 시 모델 다운로드 시간이 걸리므로 Bash `run_in_background`로 띄우고 Monitor로 진행 상황을 추적하는 것을 추천.
- 출력 파일:
  - `<audio>.txt` — `[HH:MM:SS] 문장` 줄 단위 transcript
  - `<audio>.json` — segments + 메타데이터 (audio_sha1, duration, model, device, RTF 등)
- 표준출력에 위 두 경로와 메타데이터가 JSON으로 한 번 더 찍히니, 거기서 transcript 경로를 읽어 다음 단계로 넘어간다.

### 3단계: transcript 읽기 및 정리 — 세 가지 핵심 구조

`<audio>.txt` 파일을 Read 도구로 읽고, 아래 3개 섹션으로 재구성한다.

#### 1. 논의 내용 (이번 회의에서 오간 이야기)
- 회의에서 다뤄진 주제·발언·결정·공유 내용.
- 시간 순서가 중요한 경우 자연스러운 흐름을 유지하고, 같은 주제는 묶는다.
- "누가 말했는지"는 transcript에서 확정 불가하므로 **발언자명은 적지 않는다**. 의제·내용만 남긴다.

#### 2. 문제점 (회의에서 제기된 문제·이슈·리스크)
- 명시적으로 문제로 거론된 항목. "잘 안 된다", "막혔다", "오류", "리스크", "걱정", "이해 안 됨" 등.
- 각 항목은 한두 줄로 압축. 가능하면 **주제 키워드**를 앞에 단다 (예: `[빌드 환경 차이]`).

#### 3. 해결방안 (스크럼/논의에서 나온 대응책·액션·합의)
- 문제점에 대응되는 해결책·후속 액션·결정 사항.
- 가능하면 2번 항목과 1:1 대응되도록 같은 주제 키워드로 묶는다.
- 아직 해결책이 나오지 않은 문제는 "**미해결 / 후속 논의 필요**" 표시.

### 정리 시 규칙 (엄수)

- **하이픈 `-` 불릿** 고정. 이모지·`◆` 소제목 금지.
- **체언형 종결** (`~완료`, `~수정`, `~정리`, `~필요`, `~예정` 등). 한 불릿 = 한 문장.
- **섹션당 5~10개 불릿 목표**. 비슷한 항목은 병합.
- **발언자명·`(이름)` 표기 금지** (사용자 요청: 화자 구분은 나중에 수동 정리).
- transcript의 오타·받아쓰기 오류는 문맥에 맞게 교정하되 의미를 바꾸지 않는다.
- 잡담·인사·의미 없는 추임새는 제외.
- 영문 기술 용어는 그대로 노출 (JSON, API, Dictionary 등).

### 4단계: 통합 docx 작성 (python-docx)

```
표지
  제목: "<팀명> 회의 녹음 정리 (<날짜>)"
  부제: 원본 오디오 파일명, 모델, 녹음 길이
[페이지 브레이크]

■ 1. 논의 내용
- 항목
- 항목
[페이지 브레이크]

■ 2. 문제점
[주제 키워드 1]
- 내용 ...

[주제 키워드 2]
- 내용 ...
[페이지 브레이크]

■ 3. 해결방안
[주제 키워드 1]
- 해결책 ...

[주제 키워드 2]
- 미해결 / 후속 논의 필요 — 사유 ...
```

스타일 (기존 /회의록정리와 통일):
- **한글 폰트**: 맑은 고딕. 모든 run에 `w:eastAsia`, `w:ascii`, `w:hAnsi` 3가지로 적용.
- **페이지 브레이크**: `WD_BREAK.PAGE` 사용. 3개 섹션 사이에 강제 삽입.
- **색상 계층**:
  - 표지 제목 18pt, 굵게, 가운데 정렬
  - 섹션 제목(`■ N.`) 15pt, 진한 남색 `RGBColor(0x1F, 0x4E, 0x79)`, 굵게
  - `[주제]` 블록 헤더 12pt, 밝은 파랑 `RGBColor(0x2E, 0x75, 0xB6)`, 굵게
  - 본문 불릿 `List Bullet`

### 5단계: 요약본(.summary.md) 작성

docx 옆에 **한 화면에서 회의 전체가 잡히는 마크다운 요약본**을 만든다. docx 가 정식 회의록이라면 이 파일은 슬랙·노션 등에 그대로 붙여 넣어도 좋은 "공유용 한 페이지" 다. 가독성을 위해 표·강조·계층 불릿을 적극 활용한다.

#### 파일 구조 (마크다운)

```markdown
# <팀명> 회의 녹음 요약 — <날짜>

> **TL;DR**: 회의의 결론·핵심 화두를 1~3문장(약 100~250자)으로 압축. 이 한 단락만 읽어도 회의 전체 분위기가 잡혀야 한다.

| 항목 | 값 |
|---|---|
| 녹음 길이 | HH:MM |
| 받아쓰기 | M:SS (RTF 0.XXx) |
| 모델 / 장치 | small / cpu / int8 |
| 분류 항목 | 논의 N / 문제점 M / 해결방안 K |
| 액션 아이템 | N건 (담당 미지정 m / 기한 미지정 k) |

**핵심 키워드**: `키워드1` `키워드2` `키워드3` `키워드4` `키워드5`

---

## 1. 핵심 포인트

회의에서 오간 이야기 중 가장 중요한 6~10개. 각 항목은 한 줄 요약 + 필요 시 한 줄짜리 indented 부연 (왜 중요한지·후속 영향).

- **포인트 제목 또는 핵심 동사** — 한 줄 본문
  - (선택) 한 줄 부연 — 맥락·근거·후속 영향
- **다음 포인트** — 본문
- ...

## 2. 결정 사항

명시적으로 확정된 항목만 번호 매김. "검토 중", "고민 중" 은 미해결 이슈로 보낸다.

1. **결정 제목** — 결정 내용 + 결정 이유/배경
2. **결정 제목** — 내용 + 이유
3. ...

확정된 결정이 없는 회의면 다음 한 줄로 대체: `*이번 회의에서 확정된 결정 사항 없음*`

## 3. 액션 아이템

GFM 표 형식으로 한 눈에 비교 가능하게 정리한다.

| # | 액션 | 담당 | 기한 | 상태 / 비고 |
|---|---|---|---|---|
| 1 | 액션 한 줄 설명 | ? | ? | 신규 |
| 2 | 액션 한 줄 설명 | ? | ? | 후속 논의 필요 |

> **담당·기한이 transcript 에서 확정되지 않은 경우 `?` 로 둔다.** 추정 금지. "상태 / 비고" 는 `신규` / `후속 논의 필요` / `차주 회의 안건` 등 한 단어로 끝낸다.

## 4. 미해결 이슈

다음 회의로 이월되는 항목. 각 항목에 **왜 미해결인지 + 다음 결정 시점/방법** 을 함께 적는다.

- **이슈 제목** — 미해결 사유 + 다음 회의에서 결정할 사항
- **이슈 제목** — 사유 + 결정 방법
- ...

미해결 항목이 없으면 섹션 자체를 생략한다.

## 5. 회의 맥락 (선택)

본 의제와 직접 관련은 없지만 흐름 파악에 도움이 되는 보조 정보 — 출결, 휴가, 일정 변경, 행정 공지 등. 없으면 섹션 생략.

- 항목 (출결·휴가 등)
- 항목

---
*원본 transcript: `<audio>.txt` / docx: `<팀명>_회의녹음정리_<날짜>.docx`*  
*받아쓰기: whisper-rabbit (`<model>` on `<device>/<compute_type>`, RTF 0.XXx)*
```

#### 작성 규칙 (엄수)

- **TL;DR**: 1~3 문장, 100~250자. 회의의 결론·핵심 화두를 압축. 단순 의제 나열 금지.
- **메타 표**: docx 표지 정보와 같은 출처(`<audio>.json` 메타)에서 가져온다. 직접 추정 금지.
- **핵심 키워드**: 회의를 검색할 수 있는 5~7개의 인라인 코드 태그. 한국어/영문 가리지 않음. 사람 이름은 넣지 않는다.
- **핵심 포인트**: 6~10개. 한 줄 본문 + 필요 시 1줄 부연(indented). 부연은 *왜 중요한가* 또는 *후속 영향*에 집중하고, 단순 반복은 금지.
- **결정 사항**: 번호 매김 + `**제목** — 내용 + 이유` 형식. 확정 안 된 항목은 미해결 이슈로.
- **액션 아이템**: GFM 표 5컬럼(`# / 액션 / 담당 / 기한 / 상태 비고`). 담당·기한 미확정은 `?`. 상태는 한 단어.
- **미해결 이슈**: 사유 + 다음 결정 시점/방법 함께 적기. 단순 "후속 논의 필요" 만 적지 않는다.
- **회의 맥락**: 의제와 별개의 출결·휴가·행정만 모은다. 핵심 포인트에 섞이면 안 된다.
- **발언자 표기 금지** (docx 와 동일 원칙). 사람 이름은 어디에도 적지 않는다.
- 영문 기술 용어는 그대로 노출 (JSON, API, Dictionary 등).
- 추정으로 사실을 보태지 않는다. transcript 에 없는 정보는 적지 않는다.

### 6단계: 저장 및 결과 보고

산출물 두 개를 같이 저장한다.

- **docx**: `<팀명>_회의녹음정리_<날짜>.docx` (예: `팀2_회의녹음정리_2026-05-13.docx`)
- **요약본**: `<팀명>_회의녹음정리_<날짜>.summary.md`
- 저장 위치: `C:\Users\21-01-00131\Desktop\`
- 같은 이름이 열려 있어 `PermissionError` 발생 시 `_v2`, `_v3` 증분 저장.
- **transcript txt/json 도 같이 보관**. 사용자가 나중에 화자 구분·원문 확인을 할 수 있도록 transcript 경로를 결과 보고에 명시한다.

#### 결과 보고 (간결히)

- 생성된 docx 절대경로 + 크기
- 생성된 요약본(.summary.md) 절대경로 + 액션 아이템 개수
- transcript txt/json 절대경로
- 녹음 길이, 처리 소요 시간, RTF, 사용한 모델 / device / compute_type (json 메타에서 읽어 표시)
- 섹션별 항목 개수 (논의 N / 문제점 M / 해결방안 K)
- 분류가 모호해 `[분류 확인 필요]`로 표기한 항목이 있다면 목록 안내

## 주의사항

- transcript는 받아쓰기 결과라 **오타·문장 경계 오류**가 있을 수 있다. 정리할 때는 의미 단위로 묶어서 의도가 명확해지도록 다듬되, **사실관계를 추측으로 보태지 않는다**. 들리지 않는 부분이 분류에 핵심이면 `[분류 확인 필요]`로 남긴다.
- transcript에서 발언자를 추정하지 않는다. (사용자 요청: 누가 말했는지는 나중에 사람이 직접 정리)
- 이전 회의·다른 팀 내용은 어떤 경로로도 참조하지 않는다.
- 첫 실행 시 Whisper 모델 다운로드로 몇 분~10분 이상 걸릴 수 있음을 사용자에게 미리 안내.
- 매우 긴 회의(2시간+)는 받아쓰기만 30분+ 걸릴 수 있다. `run_in_background`로 띄우고 Monitor로 진행 상황을 사용자에게 보여준다.
